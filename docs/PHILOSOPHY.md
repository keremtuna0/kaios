# KAIOS Philosophy

KAIOS is built on one principle:

> AI should accelerate software engineering, not replace software engineers.

## Thinking Before Coding

Implementation is the final step. KAIOS starts with understanding, constraints,
requirements, architecture, and trade-offs.

This protects the developer from the most common AI-assisted mistake: accepting
fast code before knowing whether it is the right code.

## Simplicity Over Complexity

KAIOS prefers the simplest design that satisfies the requirements. Abstractions
must earn their place.

Use a pattern when it removes real complexity. Avoid a pattern when it only
makes the solution look more sophisticated.

## Explicit Decisions

Every architectural decision should explain:

- what was chosen
- why it was chosen
- what alternatives were rejected
- what trade-off was accepted

This turns AI collaboration into engineering learning instead of invisible
magic.

## Modularity

Every KAIOS capability should have one responsibility. Modules communicate
through clear artifacts such as context summaries, contracts, plans, review
reports, and learning summaries.

## Production-First Mindset

Even small projects should consider security, testing, maintainability,
observability, and future change.

Production-first does not mean over-engineered. It means decisions are made
with consequences in mind.

## Developer Growth

The best KAIOS outcome is not that AI wrote more code. The best outcome is that
the developer understands the problem, solution, trade-offs, and principles
better than before.
